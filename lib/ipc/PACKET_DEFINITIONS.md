# Служебные и функциональные пакеты протокола RS485

## Классификация пакетов

### 1. Служебные пакеты (Service Packets) - 0x01-0x04

Служебные пакеты управляют коммуникацией, диагностикой и синхронизацией между master и slave.

#### 1.1 Heartbeat (0x01)

**Назначение:** Периодический сигнал жизни от slave к master

**Отправитель:** Slave (самостоятельно каждые N миллисекунд)

**Структура данных:**
```
Байт 0: STATUS      - Статус slave (битовая маска)
Байт 1-2: UPTIME_MS - Время работы slave в миллисекундах (big-endian)
```

**Флаги STATUS:**
```
Бит 7-3: Зарезервированы
Бит 2:   ERROR      - 1 = ошибка в устройстве
Бит 1:   BUSY       - 1 = slave занят выполнением команды
Бит 0:   OK         - 1 = slave готов принимать команды
```

**Пример использования:**
- Контроль живучести slave
- Определение отказа узла
- Сбор статистики загруженности

**Таймаут:** Если heartbeat не получен > 2 × heartbeat_interval → slave считается offline

---

#### 1.2 Ack (0x02)

**Назначение:** Подтверждение успешного получения пакета

**Отправитель:** Slave в ответ на успешный прием запроса от Master

**Структура данных:**
```
Байт 0: SEQ_ID - ID пакета, который подтверждается
```

**Использование:**
- Гарантия доставки в критичных системах
- Синхронизация асинхронных операций

---

#### 1.3 Nak (0x03)

**Назначение:** Отрицательное подтверждение (пакет не принят)

**Отправитель:** Slave при ошибке приема

**Структура данных:**
```
Байт 0: SEQ_ID      - ID пакета, который не принят
Байт 1: ERROR_CODE  - Код ошибки (из ErrorCode enum)
```

**Причины NAK:**
- CRC ошибка
- Невалидный формат пакета
- Buffer overflow на slave
- Slave временно не готов (busy)

---

#### 1.4 Error (0x04)

**Назначение:** Асинхронное уведомление об ошибке

**Отправитель:** Slave при возникновении критичной ошибки

**Структура данных:**
```
Байт 0: ERROR_CODE       - Код ошибки
Байт 1: ADDITIONAL_INFO  - Дополнительная информация (specific to error)
```

**Примеры:**
- Обнаружение Hardware ошибки
- Температура превышена
- Потеря синхронизации

---

## 2. Функциональные пакеты - Состояние (State Packets) - 0x10-0x12

Используются для обмена состоянием устройства между Master и Slave.

#### 2.1 State Request (0x10)

**Назначение:** Master запрашивает состояние от slave

**Отправитель:** Master

**Структура данных:**
```
Байт 0: STATE_ID - ID запрашиваемого состояния
                   0x00-0xFE = конкретное состояние
                   0xFF = запрос всех состояний
```

**Примеры STATE_ID:**
- 0x00 = Основное состояние (напряжение, ток, температура)
- 0x01 = Статус устройства
- 0x02 = Конфигурация
- 0xFF = Получить все доступные состояния

**Поведение slave:**
- Slave отправляет State Response с запрошенными данными
- Или отправляет Error если STATE_ID не поддерживается

---

#### 2.2 State Response (0x11)

**Назначение:** Slave отправляет состояние в ответ на State Request или самостоятельно

**Отправитель:** Slave

**Структура данных:**
```
Байт 0: STATE_ID    - ID состояния
Байт 1-N: STATE_DATA - Данные состояния (переменная длина)
```

**Максимальный размер:** 248 байт (1B state_id + 247B data)

**Пример структуры для STATE_ID=0x00:**
```cpp
struct State0 {
    uint16_t voltage;      // mV
    uint16_t current;      // mA
    int16_t temperature;   // °C × 10
    uint8_t flags;
} = {3300, 500, 250, 0x01};
```

---

#### 2.3 State Update (0x12)

**Назначение:** Slave самостоятельно отправляет обновление состояния без запроса

**Отправитель:** Slave (при изменении критичного параметра)

**Структура данных:** Идентична State Response

**Использование:**
- Отправить критичное предупреждение (температура высокая)
- Event notification (кнопка нажата)
- Периодические обновления важных параметров

**Различие от State Response:**
- Инициируется slave, а не master
- Не требует предварительного State Request
- Обычно отправляется для важных событий

---

## 3. Функциональные пакеты - Команды (Command Packets) - 0x20-0x22

Используются для выполнения команд на slave и получения результатов.

#### 3.1 Command Execute (0x20)

**Назначение:** Master отправляет команду к slave на выполнение

**Отправитель:** Master

**Структура данных:**
```
Байт 0: CMD_ID      - ID команды
Байт 1-N: CMD_DATA  - Параметры команды (переменная длина)
```

**Максимальный размер:** 248 байт (1B cmd_id + 247B data)

**Примеры CMD_ID:**
- 0x01 = Включить выход
- 0x02 = Выключить выход
- 0x10 = Установить напряжение
- 0x20 = Запустить калибровку
- 0xFF = Перезагрузка slave

**Пример структуры CMD_DATA для CMD_ID=0x10:**
```cpp
struct SetVoltageCmd {
    uint16_t target_voltage;  // в mV
    uint8_t ramp_rate;        // мV/ms
} = {3500, 10};  // 3.5V с рампой 10mV/ms
```

**Поведение slave:**
1. Получить команду
2. Отправить ACK (опционально)
3. Выполнить команду
4. Отправить Command Response с результатом
5. Или отправить Error если команда не поддерживается

---

#### 3.2 Command Response (0x21)

**Назначение:** Slave отправляет результат выполнения команды

**Отправитель:** Slave

**Структура данных:**
```
Байт 0: CMD_ID           - ID выполненной команды
Байт 1: STATUS           - Статус выполнения (ErrorCode)
Байт 2-N: RESPONSE_DATA  - Результат/данные (переменная длина)
```

**Максимальный размер:** 248 байт (1B cmd_id + 1B status + 246B data)

**Возможные STATUS коды:**
- 0x00 = kNoError (успешно)
- 0x03 = kCommandNotSupported (команда не поддерживается)
- 0x04 = kCommandFailed (ошибка выполнения)
- 0x06 = kBufferOverflow (переполнение)
- 0x07 = kInvalidState (неправильное состояние для команды)

**Пример Response для CMD_ID=0x10:**
```cpp
struct SetVoltageResponse {
    uint16_t actual_voltage;  // Достигнутое напряжение
    uint8_t error_code;       // 0 = OK
} = {3498, 0};
```

---

#### 3.3 Command Abort (0x22)

**Назначение:** Master отменяет текущую команду

**Отправитель:** Master

**Структура данных:**
```
Байт 0: CMD_ID - ID команды для отмены
```

**Поведение slave:**
- Если команда исполняется → отменить и отправить Command Response с kCommandFailed
- Если команда уже завершена → отправить ACK или NAK
- Если команда неизвестна → отправить Error

---

## Последовательность обмена (Message Sequences)

### Request-Response Pattern

```
Master                          Slave
  |                              |
  |-- Command Execute (seq=1) -->|
  |                          [execute]
  |                              |
  |<-- Command Response (seq=1) -|
  |     (status, result)         |
```

### Unsolicited Update Pattern

```
Master                          Slave
  |                              |
  |                         [event occurs]
  |<-- State Update (seq=5) -----|
  |     (state_id, data)         |
  |                              |
```

### Heartbeat Pattern

```
Master                          Slave
  |                              |
  |                     [every N ms]
  |<-- Heartbeat (seq=counter)---|
  |     (status, uptime)         |
  |                              |
  |-- [no response needed] ------>|
  |                              |
```

### Error Recovery Pattern

```
Master                          Slave
  |                              |
  |-- Command Execute (seq=1) -->|
  |                          [CRC error]
  |<-- NAK (error_code=0x01) -----|
  |     (seq=1, CRC error)       |
  |                              |
  |-- Command Execute (seq=2) -->| [retry]
  |                          [success]
  |<-- Command Response (seq=2) -|
```

## Таблица кодов ошибок

| Код | Enum | Описание |
|-----|------|---------|
| 0x00 | kNoError | Успешно |
| 0x01 | kCRCError | Ошибка CRC при передаче |
| 0x02 | kProtocolError | Нарушение протокола |
| 0x03 | kCommandNotSupported | Команда не поддерживается |
| 0x04 | kCommandFailed | Команда не выполнена |
| 0x05 | kTimeout | Timeout операции |
| 0x06 | kBufferOverflow | Переполнение буфера |
| 0x07 | kInvalidState | Недопустимое состояние |
| 0xFF | kUnknownError | Неизвестная ошибка |

## Рекомендации по использованию

### Для State пакетов:
- Группировать связанные параметры в одно состояние
- Использовать STATE_ID=0xFF для получения всех состояний при инициализации
- State Update отправлять только при критичных изменениях

### Для Command пакетов:
- Команда должна быть атомарной операцией
- Включить timeout на master при ожидании ответа (обычно 1-2 сек)
- Для long-running команд использовать State Updates для прогресса

### Для Heartbeat:
- Интервал 5-10 секунд типичен для стабильных систем
- Интервал < 1 сек для критичных систем с быстрым обнаружением отказа
- Использовать флаги STATUS для быстрого определения проблем

### Общее:
- Всегда проверять response status перед использованием данных
- Реализовать retry logic с exponential backoff для unreliable каналов
- Логировать все ошибки и таймауты для отладки
